## Executive Summary

Reviewed 11 form-related components (`sgds-input`, `sgds-textarea`, `sgds-select`, `sgds-checkbox`, `sgds-checkbox-group`, `sgds-radio`, `sgds-radio-group`, `sgds-file-upload`, `sgds-combo-box`, `sgds-quantity-toggle`, `sgds-switch`, `sgds-datepicker`) plus the shared infrastructure (`SgdsFormValidatorMixin`, `InputValidationController`, `FormControlElement`).

The architecture is coherent: `SgdsFormValidatorMixin` wraps every component with `static formAssociated = true`, an `ElementInternals` instance, a `formResetCallback`, and a `InputValidationController` that mirrors the internal shadow `<input>` validity into the element's internals. This is a solid foundation.

Three **critical** bugs were found that break native form submission behaviour, three **major** gaps across the whole component suite, and several accessibility and API-consistency issues at medium/minor severity.

---

## Strengths

- `SgdsFormValidatorMixin` centralises FACE boilerplate cleanly. A single mixin gives every component `formResetCallback`, `setValidity`, `checkValidity`, `reportValidity`, and `ElementInternals` wiring.
- `InputValidationController` mirrors internal-input validity into `ElementInternals` via `validateInput()` — the pattern handles all constraint-validation flags correctly.
- `noValidate` per-component escape hatch (`_mixinShouldSkipSgdsValidation`) and the ability to cancel `sgds-blur`/`sgds-change` events before validation fires are thoughtful design decisions.
- Proxy validation inputs in CheckboxGroup and RadioGroup (`<input type="text" tabindex="-1">`) allow constraint validation without a real focusable element — a clean workaround for the limitations of FACE on group controls.
- `_mixinResetFormControl()` pattern allows each component to provide bespoke reset logic that the mixin calls from `formResetCallback`, keeping extension straightforward.

## Issues by Severity

### Critical

#### C-1 — `sgds-checkbox`: unchecked checkbox submits its value to FormData

**File:** `src/components/Checkbox/sgds-checkbox.ts`

`_mixinSetFormValue()` (inherited from the mixin) calls `this._internals.setFormValue(this.value)` unconditionally. For a checkbox, the HTML spec says the field should appear in FormData only when `checked`. Because `SgdsCheckbox` has no override and no getter that returns `null` when unchecked, an unchecked `<sgds-checkbox name="opt" value="yes">` will still submit `opt=yes`.

**Suggested fix:** Override `_mixinSetFormValue` in `SgdsCheckbox`:

```ts
_mixinSetFormValue() {
  this._internals.setFormValue(this.checked ? (this.value ?? "on") : null);
}
```

---

#### C-2 — `sgds-file-upload`: no `value` property → files never reach FormData

**File:** `src/components/FileUpload/sgds-file-upload.ts`

The mixin's `_mixinSetFormValue` reads `this.value`. `SgdsFileUpload` never declares a `value` property, so `this.value` is `undefined` at runtime. Every call to `_mixinSetFormValue()` executes `this._internals.setFormValue(undefined)`, which clears the form entry. Files are completely invisible to native form submission.

**Suggested fix:** Override `_mixinSetFormValue` to build a `FormData` from `this.selectedFiles`:

```ts
_mixinSetFormValue() {
  if (this.selectedFiles.length === 0) {
    this._internals.setFormValue(null);
    return;
  }
  const fd = new FormData();
  this.selectedFiles.forEach(f => fd.append(this.name, f));
  this._internals.setFormValue(fd);
}
```

---

#### C-3 — `sgds-switch`: not form-associated at all

**File:** `src/components/Switch/sgds-switch.ts`

`SgdsSwitch` extends `SgdsElement` directly. It does not use `SgdsFormValidatorMixin`, has no `static formAssociated = true`, no `ElementInternals`, no `name` property, and does not call `setFormValue`. Placing a switch inside a `<form>` produces no FormData entry; form reset does not restore its state; constraint validation is not available.

The presence of `@defaultValue("checked")` implies reset behaviour was intended, but the infrastructure to support it is absent.

**Suggested fix:** Migrate `SgdsSwitch` to extend `FormControlElement` and apply `SgdsFormValidatorMixin`, similar to `SgdsCheckbox`. Add `name` property and override `_mixinSetFormValue` to submit `this.checked ? this.value ?? "on" : null`.

### Major

#### M-1 — `formDisabledCallback` not implemented on any component

**File:** `src/utils/validatorMixin.ts` (and all components using it)

The FACE specification defines `formDisabledCallback(disabled: boolean)` which fires when a `<fieldset disabled>` ancestor enables or disables the component. None of the SGDS form components implement this callback. As a result, wrapping SGDS form controls in a `<fieldset disabled>` will have no effect — inputs remain interactive.

**Suggested fix:** Add to `SgdsFormValidatorMixin`:

```ts
formDisabledCallback(disabled: boolean) {
  this.disabled = disabled;
}
```

Individual components that have additional side-effects on disable (e.g. QuantityToggle button states) may need to override this.

---

#### M-2 — `sgds-radio`: duplicate ARIA roles from host + native input

**File:** `src/components/Radio/sgds-radio.ts`

`setInitialAttributes()` sets `role="radio"`, `aria-checked`, and `aria-disabled` on the host element. The shadow DOM `<input type="radio">` also has its own `aria-checked` and `aria-disabled` bindings and carries the implicit role of `radiobutton`. Assistive technologies will see **two** radio button nodes for every `sgds-radio` — one from the host, one from the native input — causing duplicate announcements.

**Suggested fix:** Remove the `role="radio"`, `aria-checked`, and `aria-disabled` attributes from the host and let the native input carry all semantics. Move the `tabindex` management to the native input instead. Alternatively, add `role="presentation"` to the shadow `<input>` and keep all ARIA on the host, but the first approach is simpler and more resilient.

---

#### M-3 — `hasFeedback` prop type is inconsistent across the component suite

| Component | Type |
|---|---|
| `sgds-input` | `"style" \| "text" \| "both"` |
| `sgds-checkbox` | `"style" \| "text" \| "both"` |
| `sgds-quantity-toggle` | `"style" \| "text" \| "both"` |
| `sgds-textarea` | `boolean` |
| `sgds-select` / `sgds-combo-box` | `boolean` |
| `sgds-radio-group` | `boolean` |
| `sgds-checkbox-group` | `boolean` |
| `sgds-file-upload` | `boolean` |
| `sgds-datepicker` | `boolean` |

Consumers must know which variant each component uses. The `boolean` components always show both style and text feedback, removing the consumer's ability to show only one. This inconsistency has no rationale documented in the code.

**Suggested fix:** Standardise on `"style" | "text" | "both"` for all form components. A `boolean` `true` could be mapped to `"both"` during migration.

#### M-4 — `noValidate` property absent from most form components

**File:** All components except `sgds-input` and `sgds-textarea`

`SgdsInput` and `SgdsTextarea` expose `noValidate` to disable per-component SGDS validation. The mixin's `_mixinShouldSkipSgdsValidation()` also checks `this.noValidate`:

```ts
_mixinShouldSkipSgdsValidation() {
  const form = this.closest("form");
  return form?.noValidate || this.noValidate;
}
```

Because `noValidate` is not declared on `SgdsSelect`, `SgdsComboBox`, `SgdsCheckboxGroup`, `SgdsRadioGroup`, `SgdsFileUpload`, `SgdsQuantityToggle`, and `SgdsDatepicker`, developers cannot disable SGDS validation on these controls without wrapping the whole form in `novalidate`. `this.noValidate` evaluates to `undefined` (falsy) at runtime, so the guard silently does nothing.

**Suggested fix:** Add `noValidate` to `FormControlElement` (the shared base) so all components inherit it automatically.

---

#### M-5 — `sgds-quantity-toggle`: no `required` property

**File:** `src/components/QuantityToggle/sgds-quantity-toggle.ts`

`SgdsQuantityToggle` has `min` and `max` properties but no `required`. The inner `<sgds-input>` is rendered without `?required`, so constraint validation cannot enforce that the field is non-zero. Attempting `quantityToggle.reportValidity()` on an empty required quantity will always return `true`.

**Suggested fix:** Add `@property({ type: Boolean, reflect: true }) required = false;` to `SgdsQuantityToggle` and pass it through to the inner `<sgds-input>`.

### Medium

#### Med-1 — `<fieldset>` group labels use `<label>` instead of `<legend>`

**Files:** `src/components/Checkbox/sgds-checkbox-group.ts`, `src/components/Radio/sgds-radio-group.ts`

Both group components render a `<fieldset>` but label it with a floating `<label class="form-label">`. A `<fieldset>` must be labelled by its first `<legend>` child for screen readers to announce the group name when they enter the fieldset. A plain `<label>` without a `for` attribute inside a `<fieldset>` does not satisfy this requirement.

**Suggested fix:** Replace the `<label>` with `<legend>` inside the `<fieldset>`. The existing `form-label` CSS class can be applied to `<legend>` to retain visual style.

---

#### Med-2 — `sgds-input` / `sgds-select`: hint text in `aria-labelledby`, not `aria-describedby`

**Files:** `src/components/Input/sgds-input.ts`, `src/base/select-element.ts`

The rendered inputs include:

```html
aria-labelledby="${this._labelId} ${this._controlId}Help ..."
```

`${this._controlId}Help` is the ID of the hint text `<div>`. Adding hint text to `aria-labelledby` means it becomes part of the accessible *name* of the control, overriding the `<label>` and producing verbose combined announcements. Hint text is supplementary description, not the control name, and belongs in `aria-describedby`.

**Suggested fix:** Remove the hint-text ID from `aria-labelledby` and add a separate `aria-describedby` pointing to the hint text element (and the invalid feedback when visible):

```html
aria-labelledby="${this._labelId}"
aria-describedby="${this._controlId}Help ${this.invalid && this.hasFeedback ? `${this._controlId}-invalid` : ''}"
```

---

#### Med-3 — `sgds-textarea`: no `aria-describedby` for hint text or error messages

**File:** `src/components/Textarea/sgds-textarea.ts`

The `<textarea>` element has `aria-invalid` but no `aria-describedby` at all. Both the hint text `<div>` (`${this._controlId}Help`) and the invalid feedback `<div>` (`${this._controlId}-invalid`) are rendered in the DOM but are never programmatically associated with the textarea. Screen readers will not announce them.

**Suggested fix:** Add `aria-describedby` to the `<textarea>`, pointing to the hint text ID normally and additionally the error feedback ID when invalid:

```html
aria-describedby="${this._controlId}Help ${this.invalid && this.hasFeedback ? `${this._controlId}-invalid` : ''}"
```

---

#### Med-4 — Proxy validation inputs not hidden from assistive technologies

**Files:** `src/components/Checkbox/sgds-checkbox-group.ts`, `src/components/Radio/sgds-radio-group.ts`

Both group components render a hidden `<input type="text" tabindex="-1">` to proxy constraint validation. `tabindex="-1"` prevents keyboard focus but the element remains in the accessibility tree. Screen reader users navigating with virtual cursor may still encounter an unnamed text field.

**Suggested fix:** Add `aria-hidden="true"` to each proxy input:

```html
<input type="text" class="..." tabindex="-1" aria-hidden="true" ... />
```

---

#### Med-5 — `SgdsFormValidatorMixin`: `formStateRestoreCallback` not implemented

**File:** `src/utils/validatorMixin.ts`

`formStateRestoreCallback(state, reason)` is a FACE lifecycle hook that browsers call during autofill, bfcache restoration, and session restore. Without it, browser-saved form state is ignored and form fields appear empty after navigation. For components such as `sgds-datepicker` and `sgds-select`, this is a noticeable regression.

**Suggested fix:** Add a default `formStateRestoreCallback` to the mixin:

```ts
formStateRestoreCallback(state: string | File | FormData | null, _reason: "autocomplete" | "restore") {
  if (typeof state === "string") {
    this.value = state;
  }
}
```

Components with non-string state (FileUpload, DatePicker, ComboBox multi-select) will need component-level overrides.

#### Med-6 — `SgdsCheckboxGroup`: `_isTouched` requires *all* checkboxes to blur

**File:** `src/components/Checkbox/sgds-checkbox-group.ts`

```ts
this.addEventListener("sgds-blur", e => {
  this._blurredCheckboxes.add(checkbox);
  if (Array.from(this._blurredCheckboxes).length === this._checkboxes.length) {
    this._isTouched = true;
    ...
  }
});
```

Validation is not triggered until every item in the group has individually received and lost focus. Standard UX marks a control as touched when *any* interaction occurs. A user who clicks one checkbox and Tab-navigates away without visiting all boxes will never see validation feedback.

**Suggested fix:** Mark `_isTouched = true` on the first `sgds-blur` event, consistent with the behaviour of `SgdsInput`, `SgdsTextarea`, and `SgdsRadioGroup`.

---

#### Med-7 — `sgds-select._handleInputBlur`: `e.preventDefault()` on a non-cancellable event

**File:** `src/components/Select/sgds-select.ts`

```ts
protected async _handleInputBlur(e: Event) {
  e.preventDefault();
  this.emit("sgds-blur");
}
```

`blur` events are not cancellable; calling `preventDefault()` is a documented no-op. This is misleading and suggests the author intended to prevent some side effect that simply cannot be cancelled via this mechanism.

**Suggested fix:** Remove `e.preventDefault()`.

### Low / Minor

#### L-1 — `sgds-file-upload` JSDoc: wrong event detail access path

**File:** `src/components/FileUpload/sgds-file-upload.ts`, line 17

The JSDoc comment states:

The correct path is `event.detail`. `event.target` refers to the element, not the event payload.

#### L-2 — `sgds-switch`: `@defaultValue("checked")` on a non-form-associated component

**File:** `src/components/Switch/sgds-switch.ts`

`@defaultValue("checked")` documents intent to reset on form reset, but because `SgdsSwitch` is not a FACE, the `formResetCallback` never fires. The decorator stores `defaultChecked` but it is never used for actual reset. This is misleading for anyone reading the component source.

---

#### L-3 — `sgds-combo-box`: `_isTouched` is declared but never set to `true`

**File:** `src/components/ComboBox/sgds-combo-box.ts` (inherits `_isTouched` from `SelectElement`)

```ts
if (!this._isTouched && this.value === "") return;
```

`_isTouched` is never assigned `true` anywhere in `SgdsComboBox` or `SelectElement`. The guard therefore reduces at runtime to `if (this.value === "") return`. The variable is vestigial dead state.

**Suggested fix:** Either remove `_isTouched` and write the condition as `if (this.value === "") return;`, or properly set `_isTouched = true` when the control loses focus.

---

#### L-4 — `sgds-quantity-toggle`: clamping condition incorrect when `step > 1`

**File:** `src/components/QuantityToggle/sgds-quantity-toggle.ts`

```ts
if (parseInt(sgdsInput.value) < this.step || sgdsInput.value === "") {
  sgdsInput.value = "0";
}
```

When `step=2` and the user manually enters `1`, `parseInt("1") < 2` is `true`, so the value is silently clamped to `0`. A value of `1` is valid for a step-2 counter starting at `0`; the spec requires only that submitted values are valid relative to the step baseline, not that every intermediate typed value must be a multiple of step. This also contradicts the behaviour when the user presses the decrement button (which does not apply this clamp).

**Suggested fix:** Restrict the clamp to genuinely unparseable input only:

```ts
if (sgdsInput.value === "" || isNaN(parseInt(sgdsInput.value))) {
  sgdsInput.value = "0";
}
```

---

#### L-5 — `_mixinResetFormControl` declared `private` conflicts with mixin's `declare`

**Files:** `src/components/Checkbox/sgds-checkbox.ts`, `sgds-checkbox-group.ts`, `sgds-radio-group.ts`, `sgds-quantity-toggle.ts`, `sgds-datepicker.ts`, `sgds-file-upload.ts`

The mixin class body contains `declare _mixinResetFormControl: () => void;` (a public member declaration). All implementing classes define it as `private`. TypeScript's structural typing means `private` is compile-time-only; the mixin's runtime `if (this._mixinResetFormControl)` check will find it correctly. However, some editors and strict TypeScript configurations will flag the `private` method as incompatible with the mixin's public declaration, causing developer confusion and potential CI failures in stricter projects consuming the library.

**Suggested fix:** Change `private _mixinResetFormControl()` to `protected _mixinResetFormControl()` in all implementing classes to align with the mixin's expectation.

---

#### L-6 — `sgds-input._handleBlur`: redundant double validity check on first touch

**File:** `src/components/Input/sgds-input.ts`

On first blur, `setInvalid(!this._mixinCheckValidity())` is called directly, and then `this._isTouched = true` triggers the `@watch("_isTouched")` handler which calls `setInvalid(!this._mixinCheckValidity())` a second time. This is benign but wasteful — validity is evaluated twice on every initial blur.

**Suggested fix:** Remove the direct `setInvalid` call from `_handleBlur` and rely entirely on the `_handleIsTouched` watcher, which already does the same work:

```ts
protected _handleBlur() {
  const sgdsBlur = this.emit("sgds-blur", { cancelable: true });
  if (this._mixinShouldSkipSgdsValidation()) return;
  if (sgdsBlur.defaultPrevented) return;
  this._isTouched = true;
}
```

## Language-Specific Observations (TypeScript / LitElement)

- The mixin type declaration uses `declare value: string;` to enforce that all classes provide a `value` accessor. `SgdsFileUpload` violates this contract silently because TypeScript resolves the `declare` from `FormControlElement`'s inherited `name` property rather than from the mixin's explicit declaration. Stronger enforcement (e.g. an abstract base class with `abstract value: string`) would surface this at build time.
- Several components use `queryAsync` to await shadow DOM children (`sgds-input`, `sgds-datepicker-input`). If the child component is not registered (e.g. missing `customElements.define` call), `queryAsync` will resolve `null` and subsequent property access will throw at runtime. Adding null guards or constructing with dependency injection would be safer.
- The `_mixinResetFormControl` override pattern is convention-based rather than type-enforced. Renaming the method or changing its signature in a subclass would silently break form reset without a TypeScript error.
- `SgdsSwitch` is tested against `SgdsElement` typings but its usage context (inside forms) is more complex. It should implement `SgdsFormControl` interface to make the typing gap visible at compile time.

## Recommendations (prioritised)

1. **Fix `SgdsCheckbox._mixinSetFormValue`** — override to return `null` when unchecked. This is a one-line fix with high impact.
2. **Fix `SgdsFileUpload._mixinSetFormValue`** — build a `FormData` from `selectedFiles`. The public `files` getter can be reused here.
3. **Migrate `SgdsSwitch` to FACE** — add `static formAssociated`, `ElementInternals`, `name` property, and form value management.
4. **Implement `formDisabledCallback`** in `SgdsFormValidatorMixin` — a single added method protects all components.
5. **Standardise `hasFeedback`** to `"style" | "text" | "both"` across all form components in the next major version. Boolean `true` maps to `"both"` for compatibility.
6. **Add `noValidate`** to `FormControlElement` base so it's available on every subclass.
7. **Fix ARIA on `sgds-radio`** — remove redundant host ARIA attributes and let the native input carry role semantics.
8. **Fix fieldset group labels** — swap `<label>` for `<legend>` in `SgdsCheckboxGroup` and `SgdsRadioGroup`.
9. **Fix `aria-describedby`** placement for `sgds-input`, `sgds-select`, and `sgds-textarea`.
10. **Add `required` to `SgdsQuantityToggle`**.

## Questions for Discussion

1. **`SgdsCheckboxGroup` / `SgdsRadioGroup` proxy input:** The proxy `<input type="text">` approach works but requires manually dispatching `InputEvent` to trigger constraint validation. Would a `<slot name="input">` or a dedicated `reportValidity` API surface be cleaner for consumers?
2. **`SgdsFileUpload` FormData shape:** A multi-file `FormData` with multiple entries for the same `name` is the HTML standard, but is there a preference for a single `file-name1,file-name2` string value instead? This needs to be decided before implementing C-2's fix.
3. **`hasFeedback` string union backwards-compat:** Changing `hasFeedback: boolean` to `hasFeedback: "style" | "text" | "both"` is a breaking change. Should this be gated on a new major version, or provided as an opt-in `feedbackMode` alias?
4. **`SgdsSwitch` form participation:** Is the Switch component intended for form use (e.g., persisting a "toggle this setting" boolean)? If not, documenting the non-participation explicitly would prevent confusion.
5. **`formStateRestoreCallback`:** Browser autofill/restore is complex for a datepicker (masked input) and multi-select combobox. What is the expected prioritisation of this feature relative to other roadmap items?
